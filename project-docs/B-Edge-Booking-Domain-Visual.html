Slot locked for 10 minutes. GIST constraint ensures first-write-wins. If two customers tap simultaneously, one succeeds instantly — the other gets SLOT_UNAVAILABLE.
3

Fills in special requests (optional). Submits. Artist receives a WhatsApp notification immediately.
4

Artist reviews in their dashboard. Approves or rejects. No time limit on pending duration.
5
Pay deposit → approved
Transfer deposit via Wish Money before the 48-hour deadline. Artist verifies and marks deposit received → deposit_paid. Artist then confirms → confirmed.
6
Booking confirmed ✓
Both parties receive WhatsApp confirmation. System automatically sends 24h and 2h reminders.
7
Attend → completed
Artist performs service and marks completed. Customer receives a review link via WhatsApp.
🎨 Artist journey
📱 WhatsApp notification arrives
"New booking request from [Customer]" — includes service name, time, and any special requests.
✅ Approve or reject
Approve → system sets deposit_deadline = appointment_time − 48 hours. Customer is notified via WhatsApp with the exact deadline and deposit amount.

Reject → booking becomes cancelled. Slot freed immediately. Customer notified.
💰 Deposit verification queue
Artist sees pending deposits in their dashboard. Verifies transfer in Wish Money app. Taps "Mark deposit received" (deposit_paid). Then taps "Confirm booking" (confirmed).

Two-step process prevents accidental confirmations.
📅 Day of appointment
Perform the service. Mark as completed, no-show, or cancel from the dashboard.
⏰ If deposit deadline passes
Background job runs every minute. Booking auto-transitions to expired. Slot released immediately. Waitlist triggered. No action required from artist.
3
Cancellation Policy
The 24-hour cancellation window is configurable per salon. Default is 24 hours before the appointment.
Customer cancels · >24h before
Deposit: Full refund
Status: refund_due
Artist must transfer via Wish Money. System tracks through refund_due → refunded. Customer gets WhatsApp confirmation when refund is sent.
Customer cancels · <24h before
Deposit: Forfeited
Status: cancelled
Artist keeps the full deposit. No refund is issued. Customer notified immediately via WhatsApp.
Customer no-show
Deposit: Forfeited
Status: no_show
Artist marks no-show in dashboard. Deposit kept. No-show recorded for analytics. Customer receives no refund.
Artist cancels · any time
Deposit: Full refund — always
Status: refund_due
Artist always bears the cost of cancelling. Customer receives WhatsApp with refund notice and amount. No exceptions.
Deposit deadline passes
Deposit: N/A — never paid
Status: expired
Background job auto-transitions. No deposit was ever paid. Slot freed immediately. Waitlist triggered.
No deposit required
Deposit: N/A
Status: cancelled
Service had deposit_amount = 0. No financial impact. Slot freed immediately on cancellation.

4
Slot Availability Algorithm
Runs every time a customer requests available slots. All 7 steps must pass for a slot to appear. If any step fails, the slot is excluded.
Step 1
Check store is open
Fetch business_hours for the day_of_week. Check business_hours_exceptions for holidays on that exact date.

→ Store closed: return empty list
→ Holiday (is_closed = true): return empty list
→ Exception with custom hours: use those instead
business_hours · business_hours_exceptions
Step 2
Apply same-day minimum notice
If the requested date is today: earliest slot = NOW() + same_day_notice_hours.

Default: 4 hours. Configurable per store.

Prevents booking a 2-hour makeup session that starts in 30 minutes.
stores.same_day_notice_hours
Step 3
Build blocked ranges from existing bookings
Fetch all bookings for this artist on this date with a blocking status:

held · pending · approved · deposit_paid · confirmed

Each booking = blocked time range. New slot cannot overlap any of these.
bookings WHERE status IN blocking_statuses
Step 4
Apply travel buffer for cross-store bookings
Fetch artist bookings at OTHER stores on this date.

Weekday (Mon–Thu): block 150 minutes before + after
Weekend (Fri–Sun): block 90 minutes before + after

Example: Tripoli ends 9:10am on a weekday → no Beirut slots before 11:40am.
artist_store_buffers · bookings (other stores)
Step 5
Apply processing gap support
For services with active_duration_min (kreatin, hair color):

• CHAIR blocked for full service duration
• ARTIST only blocked for active_duration_min
• Artist can take another client during processing gap
services.active_duration_min
Step 6
Flag early bird slots
If the store has early_bird_cutoff set:

• Slots before cutoff → is_early_bird = true
• Early bird slots carry an additional early_bird_fee
• Still offered — just priced differently

Beirut default: before 9:00 AM. Tripoli: before 7:30 AM.
stores.early_bird_cutoff · stores.early_bird_fee
Step 7 — Final
Generate valid slots
Iterate through the store open window in service_duration_min increments. For each candidate slot [start, end):
start ≥ earliest_allowed_time (after minimum notice)
end ≤ store closing time
[start, end) does not overlap any blocked range
start + service_duration ≤ closing time

If all pass → add to available list. Tag as early_bird if applicable.
🔒 GIST constraint — the final atomic guard
Even after all 7 steps pass, the GIST exclusion constraint on the bookings table is the last line of defence.

If two customers race for the same slot at exactly the same moment, only one INSERT succeeds. The second receives SLOT_UNAVAILABLE instantly. No application-level check can provide this guarantee — only the database constraint can.
Output: TimeSlot[] with start_time, end_time, is_early_bird, early_bird_fee
